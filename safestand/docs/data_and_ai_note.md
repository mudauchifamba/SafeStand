# SafeStand — Data & AI Usage Note (model card)

The ToR's critical rule is that forced AI is heavily penalised. SafeStand's design
principle is the same rule stated positively: **AI where judgment of unseen input is
required; databases where an authority has already answered; deterministic code
where the question has an exact answer.** Every verdict reason is labelled by which
tier produced it.

## 1. Data inventory & provenance

| Dataset | Type | Source & rights | Personal data? |
|---|---|---|---|
| `synthetic_training.csv` (900 rows) | Synthetic text documents | Generated by our disclosed script (`ml/generate_synthetic.py`), stamp-concept augmented; schema per `ml/DATA_CONTRACT.md` | None — all names/refs fictitious |
| `real_eval.csv` (17 rows) | Held-out evaluation set | Text versions of our fictitious specimen PDFs; **never trained on** (contract rule 2) | None |
| Specimen PDFs (`assets/specimens/`, `test_docs/`) | Test documents | Authored by the team, watermarked "SPECIMEN — FICTIONAL" | None |
| `known_cases.json` | High-risk areas & schemes | Compiled from cited public sources (news reporting, ZimLII judgments, ZACC statements); every entry carries its source string | No private individuals; named schemes only with public citations |
| `red_flag_rules.json` | Rule definitions | Authored from documented fraud patterns; weights sum-normalised | None |
| `gazetteer.json` | 30 suburb centroids | Approximate centroids compiled by the team; used only for coarse distance checks | None |
| `wetlands.json` | 12 documented wetlands | Indicative circles compiled from public documentation (Ramsar listing, EMA statements, press); `_meta` carries the disclaimers | None |
| User inputs (photos, documents, pins) | Runtime only | Processed on-device; sent to the vision API only on explicit user action; never stored server-side (no server exists) | Potentially — see §4 |

**Known data limitations (disclosed):** training data is synthetic (bootstrap by
design — the honest metric is out-of-distribution accuracy on held-out real-style
specimens); wetland boundaries are indicative, not survey-grade; gazetteer is
greater-Harare-first; known-cases seed set is small and grows via moderated,
source-cited additions.

## 2. AI methods, justification, validation

### AI-1 — OCR (Google ML Kit, on-device)
- **Task**: read document text (incl. stamp text) from photos. **Why AI**: reading
  arbitrary photographed documents is a perception task with no rule-based equivalent.
- **Validation**: extracted text is shown to the user before scoring (human check of
  read quality); specimen PDFs exercise the pipeline end-to-end.

### AI-2 — Document fraud classifier (ours; the core AI contribution)
- **Method**: TF-IDF (1–2 grams, sublinear tf, min_df 2) + balanced logistic
  regression. Trained in Python (`ml/train.py`) or pure Dart
  (`tool/train_model.dart`); exported to JSON; pure-Dart on-device inference.
- **Why AI over rules**: the rule engine catches *known* keyword patterns; the
  classifier learns the statistical boundary of how fraudulent vs genuine documents
  *talk*, generalising to unseen phrasing (incl. stamp content signals). The rules
  remain in the loop as the explainability layer — model decides, rules explain.
- **Why this model over larger ones**: ~50 KB, offline, milliseconds per inference
  on low-end phones, and every prediction decomposable into per-term contributions
  (`topContributions`) — explainability is a hard requirement when the output
  influences someone's life savings. A deep model would be slower, opaque, and
  unjustifiable on this data volume.
- **Validation**: **17/17 on held-out real-style specimens never seen in training**;
  the eval set is contractually never trained on; the Dart trainer refuses to export
  any model scoring below 100% on it; CI test (`fraud_classifier_test.dart`)
  re-verifies the shipped export on every run, plus a stamp-signal behavioural test.
- **Failure modes (disclosed)**: fraud phrasings far outside the training
  distribution may score mid-range — which the score blend (60% model / 40% rules)
  and the area/database checks backstop.

### AI-3 — Vision LLM (Llama-4-Scout via Groq; online, optional)
- **Tasks**: (a) classify satellite land cover at the pinned stand + vlei
  indicators; (b) judge seller photos: what they show, terrain class, authenticity
  tells (screenshots, watermarks, renders). **Why AI**: no database says what is at
  an arbitrary coordinate *today*, and no formula judges photo content — both are
  perception judgments of never-before-seen input.
- **Guardrails**: temperature 0; strict JSON output contract with whitelisted enum
  values (anything else parses to `unknown`/ignored); conservative scoring
  (agreement is a zero-weight note; only contradictions and hazards add risk);
  unavailable/failed calls are ignored entirely rather than guessed.
- **Blind cross-examination**: the two analyses never see each other's inputs; the
  *app* compares their terrain classes with a tested compatibility table. AI
  testifies; deterministic code cross-examines.
- **Known failure we found and fixed**: algae-green reservoir water misread as
  vegetation → fixed with a zoomed-out context tile + explicit green-water prompt
  guidance. Documented as a local-context adaptation.

### Deliberately NOT AI
- **Wetland & known-cases checks**: database lookups with citations — certainty
  where it exists must not be replaced by inference.
- **EXIF/pin geometry**: haversine distance — exact answers need no model.
- **Cross-examination & scoring**: deterministic, unit-tested Dart.

## 3. Human oversight

- The app never acts on a user's behalf; every output is advice with cited reasons,
  and every verdict routes to human authorities (Deeds Registry, Surveyor-General,
  Registrar of Cooperative Societies, EMA).
- A GREEN verdict is explicitly framed as "no known warning signs," never as safety.
- AI-derived reasons are labelled as AI readings with confidence, and marked
  fakeable/weak where applicable (e.g. EXIF matches never lower risk).
- Pilot: users can dispute any AI reading via feedback; disputed outputs are logged
  for team review (the oversight loop).

## 4. Personal data & consent posture

- v1 collects **no accounts and no server-side data** (no server exists).
- Seller photos may contain personal data; they are processed on-device, and sent to
  the vision API **only** on an explicit user tap, stated at the point of use. A
  consent notice at that point is in the pilot checklist.
- The bundled databases contain no private individuals; named schemes appear only
  with public citations (defamation control).
